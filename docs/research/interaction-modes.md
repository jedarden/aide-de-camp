# Interaction Modes

## Layer Separation

The realtime voice model (gpt-realtime, or equivalent) is **not** responsible for
doing work. Its job is conversational: receive speech, understand intent, hand off
to backend workers, narrate results back in audio mode. It stays thin and fast.

The actual work — status queries, coding tasks, context fetching, synthesis — is
done by backend NEEDLE agents. A separate **UI-regeneration agent** is responsible
for translating worker results into visual components on the canvas. The voice model
never touches the canvas directly.

```
Voice model       → understands intent, manages conversation flow, narrates audio
NEEDLE workers    → do the actual work (status, coding, research, context fetch)
UI-regen agent    → translates worker results into canvas components
```

### Tool Calls Are Triggers, Not Queries

Realtime model tool calls have hard timeouts — insufficient for research tasks,
context fetching, or UI rendering. Tool calls therefore **fire and return
immediately**; they register work, they don't wait for it.

```
voice model calls dispatch_intent({project: "options-pipeline", type: "status"})
  → backend enqueues the intent, returns {ack: true, intent_id: "..."}  ← immediate
  → voice model tells user "checking that now"
  → NEEDLE worker runs async
  → result arrives out-of-band: SSE push to canvas, or follow-up to voice session
```

Every tool the voice model calls is a trigger. Results come back via push,
never via the tool return value. This means no tool call blocks, regardless
of how long the underlying work takes.

## Two Distinct Surfaces

### Mobile / Audio Mode

Context: on the go, no screen available. Walking, driving, between tasks.

- **Input**: continuous voice or push-to-talk
- **Output**: synthesized speech — the AI talks back
- **Content constraint**: audio can't display a table, diff, or status grid.
  Card specs are narrated as summaries, not rendered visually.
- **Latency expectation**: conversational (2-5s response)
- **Session model**: ambient — user can trail off, pick up later

Voice output **is** the interface here. Without it the mode doesn't work.

### Multimedia / Canvas Mode

Context: at a desk, browser open, window of some size.

- **Input**: voice + text
- **Output**: dynamic HTML cards rendered in a web canvas, sized to the window
- **Content freedom**: rich components — status grids, diffs, timelines, bead lists,
  kubectl output, arbitrarily structured HTML
- **Latency expectation**: cards stream in as they resolve (first card within ~3s)
- **Session model**: persistent panel — cards accumulate, can be pinned or dismissed

Voice output has no value here. The cards are the response.

## Shared Backend, Different Output Renderers

Both modes hit the same router and the same NEEDLE workers. The difference is
in the last mile — and specifically in what generates the output:

```
utterance (voice or text)
        │
        ▼
Voice model (thin — intent extraction + narration only)
        │
        ▼
Router + NEEDLE workers (do the actual work)
        │
        ▼
   result payload
   {title, type, data, summary, ...}
        │
        ├─── [audio mode] ──────→  voice model narrates `summary` back
        │
        └─── [canvas mode] ─────→  UI-regen agent generates component HTML
                                    → renders in web canvas
```

### The UI-Regeneration Agent

In canvas mode, worker results don't render via static templates. A dedicated
agent receives the result payload and generates the appropriate visual component
for the content — choosing layout, structure, and fidelity based on what the
data actually is. A kubectl pod table looks different from a git log summary,
which looks different from a bead dependency graph.

The UI-regen agent runs after each NEEDLE worker resolves. It's the last step
before the canvas updates. Its output is HTML+CSS (and optionally JS) for a
self-contained component that slots into the canvas grid.

This means the canvas is generative, not templated. New content types don't
require new template code — the UI-regen agent figures out how to display them.

### Audio Mode Output

The voice model reads the `summary` field from each result and narrates it.
No UI-regen agent is involved. The voice model is already in the loop (it
received the utterance), so narrating the result is a natural continuation of
the same session — it doesn't require a separate TTS invocation.

The `summary` field is generated by each NEEDLE worker alongside its data
payload. One sentence per card, written to be spoken not read.

## Proactive Result Surfacing

Results don't always arrive while the user is waiting. The user speaks, continues
rambling on other topics, and the backend resolves asynchronously. When a result
lands, the voice model should be notified and surface it **at an appropriate time**
— not interrupting mid-sentence, but not silently discarding it either.

```
User speaks: "has the options pipeline caught up?"
Voice model: dispatches intent, responds "checking that now"
User continues: "also what's the ibkr-mcp status — I think I need to..."
                     ↑ user is mid-sentence
Backend: options-pipeline result arrives
                     ↓ voice model receives notification, queues it
User finishes: "...restart the feed"
Voice model: natural pause detected
             "The options pipeline came back — it's current, last event 4 minutes ago.
              Also checking IBKR MCP now."
```

### Injection Mechanism

Results re-enter the voice session as injected events — a server-side message
pushed into the realtime session that the model receives alongside the live
conversation. The model's system prompt instructs it how to handle these:

- Don't interrupt a user utterance in progress
- Surface at the next natural pause
- Prioritize by urgency (error/crash > status > informational)
- If multiple results are queued, batch them into one narration pass
- If the session has been idle for N minutes, surface immediately

### Urgency Tiers

| Tier | Examples | Behavior |
|------|----------|----------|
| Critical | worker crash, cluster unreachable, bead blocked | Surface immediately, may interrupt |
| Status | pipeline caught up, deploy complete, bead closed | Surface at next pause |
| Informational | research done, naming candidates ready | Surface at next pause or on request |
| Background | scheduled digest, low-priority updates | Batch, surface when idle |

The NEEDLE worker sets the urgency tier in the result payload. The voice model
uses it to decide when to speak.

### Canvas Mode Equivalent

In canvas mode, the card appears on the canvas as soon as the result arrives —
no voice model involvement needed. But the voice model can optionally acknowledge
it: a brief "options pipeline result is in" spoken when the card renders, giving
the user audio confirmation without requiring them to look at the screen.

This bridges the modes: a user who started a query in audio mode, then sat down
at their desk, gets both the card on the canvas and a spoken acknowledgment.

### Session Continuity

If the voice session is closed when a result arrives (user put down the phone,
closed the browser tab), the result falls back to Telegram — the same push
channel used for HUMAN bead notifications. The user picks it up when they return.

## Audio Mode Implementation Options

### Option A: Telegram voice messages
The `telegram-claude-bridge` already exists and handles bidirectional messages.
Telegram supports voice message input natively. The bridge receives a voice
message, forwards audio to whisper-stt for transcription, routes to the stream-
dispatch backend, and replies with a Telegram voice message (TTS of the summary).

Zero new infrastructure. Mobile mode is just a Telegram chat. Works on any
phone with Telegram installed.

Tradeoff: Telegram is asynchronous by nature. Doesn't feel as immediate as a
live call. But for "ramble while walking" this is fine — the user doesn't need
to hold the microphone open.

### Option B: Mobile web page (push-to-talk)
A simple, minimal HTML page — one button, no visual output. Tap to speak, release
to send. Response plays as audio via Web Speech API or a TTS endpoint. Works in
any mobile browser, no app install.

Works well for "can't see a screen" use cases but requires an open browser tab.

### Option C: Asterisk / phone call
The nixos-asterisk PBX is already deployed. A dedicated extension that patches
into the stream-dispatch backend: call the number, speak, get a voice response.

Works entirely off-screen, no app, no browser. Closest to a true "ambient"
interface. The cost is Asterisk dialplan work to bridge a call leg into the
stream-dispatch audio pipeline.

**Most likely starting point**: Option A (Telegram), since the bridge already
exists. Option B (mobile web) as the cleaner long-term solution.

## Canvas Mode: Responsive Card Rendering

The web canvas sizes cards to the window. A few natural layouts:

```
Narrow (mobile browser / side panel)     Wide (desktop, full screen)
┌──────────────────────┐                 ┌────────────┐ ┌────────────┐ ┌────────────┐
│ options pipeline     │                 │ options    │ │ ibkr-mcp   │ │ pdftract   │
│ [status card]        │                 │ pipeline   │ │ status     │ │ naming     │
├──────────────────────┤                 │ [status]   │ │ [status]   │ │ [list]     │
│ ibkr-mcp status      │                 └────────────┘ └────────────┘ └────────────┘
│ [status card]        │
├──────────────────────┤
│ pdftract naming      │
│ [list]               │
└──────────────────────┘
```

Cards are web components. Their width adapts to available columns. Type determines
internal layout (a status card looks different from a diff card or a list card).

The canvas is the DUCK-E frontend, extended with a card grid layout and a component
library for each card type.

## DUCK-E Fit Per Mode

| | Mobile / Audio | Multimedia / Canvas |
|---|---|---|
| DUCK-E WebRTC voice | ✅ Voice input | ✅ Voice input |
| DUCK-E WebSocket | Replaced by Telegram (Option A) | ✅ Card stream |
| DUCK-E voice output | ✅ TTS response | ❌ Not needed |
| DUCK-E memory | ✅ Session context | ✅ Session context |
| DUCK-E frontend | ❌ No screen | ✅ Extended as card canvas |
| DUCK-E backend | ✅ FastAPI/WebSocket base | ✅ FastAPI/SSE base |

The backend is common. The frontend diverges: audio mode has no UI (or a minimal
push-to-talk button); canvas mode has the full card grid.

## Result Schema

Results are stored. Cards are not — they are generated from results at render time.

```json
{
  "result_id":   "...",
  "intent_id":   "...",
  "topic_id":    "...",
  "session_id":  "...",
  "data":        { ... },
  "summary":     "Options pipeline is current. Last processed event 4 minutes ago.",
  "urgency":     "normal",
  "timestamp":   "2026-05-18T..."
}
```

`summary` is what the voice model narrates. `data` is the structured content the
UI-regen agent uses to generate a card appropriate for the current surface, window
size, and number of active topics. The same result produces a compact chip when
six topics are active and a detailed grid when it's the only one. Cards are
ephemeral rendering artifacts; results are the record.
