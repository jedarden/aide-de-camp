# Async Locking Mechanisms for Python / FastAPI — Research

**Bead:** adc-4783 (child of adc-50ld — "Design thread-safety approach for async FastAPI")
**Scope:** Document every viable locking/atomicity mechanism for async Python, with code, pros/cons, and scenario-based recommendations, grounded in this codebase (`src/session/store.py`, `src/sse/broadcaster.py`, the `TelegramFallback` use case).
**Status:** Research reference — feeds the design decision in adc-50ld.

---

## TL;DR / Correction to the parent doc

The adc-50ld design doc concludes that **atomic operations are "NOT POSSIBLE"** in Python and that `asyncio.Lock` is the only viable mechanism. That conclusion is **too narrow** — it holds for *in-process memory flags*, but not for the codebase's actual shared state, which lives in **SQLite** (`src/session/store.py`, already running `PRAGMA journal_mode=WAL`). SQLite transactions *are* atomic operations, and they are strictly stronger than `asyncio.Lock` for persisted state because:

- They work **across processes** (multi-worker uvicorn/gunicorn) — `asyncio.Lock` does not.
- They are **non-blocking to the event loop** entirely (the lock lives in the SQLite layer / a worker thread, not on the serving loop).
- They survive restarts; in-memory locks do not.

The right mental model is **pick the mechanism that matches where the state lives**, not "always reach for `asyncio.Lock`." See the [decision matrix](#decision-matrix).

## The concurrency model — where races actually come from

"Async FastAPI" is not one execution model. Three layers can run concurrently, and the right lock depends on which layer your state is shared across:

| Layer | Concurrency source | In-memory `asyncio.Lock` works? |
|---|---|---|
| **Event loop** (cooperative) | Coroutines yield at every `await` | ✅ Yes — this is its native use |
| **Threadpool** (`def` routes, `run_in_threadpool`, sync libs like the sync parts of `httpx`/`sqlite3`) | Preemptive OS threads | ❌ No — need `threading.Lock`, or atomics |
| **Processes** (`uvicorn --workers N`, gunicorn) | Separate interpreters | ❌ No in-memory lock at all — need DB/Redis/file atomics |

A pure `async def` route on a single-worker uvicorn (the adc deployment: `uvicorn src.main:app`, one process) only has **event-loop concurrency**. There, `asyncio.Lock` is correct and complete. But the moment a code path touches a `def` endpoint, a threadpool sync function, or a second worker, in-process locking alone is insufficient. Design for the layer that actually exists now, but prefer **atomic operations** for anything that is *also* persisted — it is correct under all three layers at once.

> **GIL caveat:** CPython's Global Interpreter Lock makes *single bytecode* operations atomic, but does **not** make read-modify-write sequences atomic (see [GIL atomicity](#mechanism-5-atomic-operations)). The GIL protects against *some* thread races but is not a real lock and must never be relied on for correctness of compound operations.

## Mechanism 1: `asyncio.Lock`

### Behavior (verified on CPython 3.12 — see appendix)

- **Non-reentrant.** Acquiring an already-held lock from the same task **deadlocks** (verified: `wait_for(lock.acquire())` times out). There is no `RLock` equivalent in asyncio. If two of your methods both take the lock and one calls the other, you must factor out an unlocked private helper.
- **Fair-ish, FIFO.** Waiters are queued in acquisition order; a lock is handed to the next waiter on release.
- **Not loop-bound on modern Python.** The `loop=` parameter was removed in 3.10. A lock created in one `asyncio.run()` can be used in another without error (verified). In practice you should still create locks on the serving loop (via lifespan — see [FastAPI patterns](#fastapi-specific-concurrency-patterns)), because *using* a lock from a task on a different loop than its waiters causes confusion even if it doesn't raise.
- **Exception-safe with `async with`.** If the body raises, the lock is released (it's a context manager). Manual `await lock.acquire()` requires `try/finally`.
- **Cancelled waiters don't corrupt state.** A task waiting on `lock.acquire()` can be cancelled cleanly.

### Usage

```python
import asyncio

class TelegramFallback:
    def __init__(self) -> None:
        self._has_logged_first_failure = False
        self._failure_count = 0
        self._first_failure_ts = None
        self._lock = asyncio.Lock()  # guards the four fields above as one unit

    async def _handle_send_failure(self, error_context: str = "") -> None:
        async with self._lock:  # serializes check-then-act across concurrent awaits
            if not self._has_logged_first_failure:
                logger.warning("First Telegram send failure: %s", error_context)
                self._has_logged_first_failure = True
                self._first_failure_ts = datetime.now()
                self._failure_count = 1
            else:
                logger.debug("Repeated failure #%d", self._failure_count + 1)
                self._failure_count += 1
```

### Pros

- ✅ Native to async; non-blocking — waiters are suspended, the loop keeps serving other requests.
- ✅ Cheap when uncontended (sub-microsecond acquire in the fast path).
- ✅ `async with` is exception-safe and readable.
- ✅ Groups multi-field state into one consistent critical section (e.g. the four first-failure fields stay in sync).

### Cons

- ❌ **Single-process only.** Zero protection across uvicorn workers or after a restart.
- ❌ **Single-loop only.** A lock created before the loop starts is awkward; mixing it with threadpool code silently fails (the thread can't `await`).
- ❌ **Non-reentrant** — easy footgun when refactoring (method calls method, both acquire).
- ❌ Provides **no durability** — if the process crashes between "set flag" and "persist it," the in-memory truth is gone.
- ❌ Holds state in the serving loop: a long critical section (e.g. doing I/O *inside* the lock) serializes every contending request.

### When to use

Guards **ephemeral, in-process, single-loop** state that is read/modified by `async def` coroutines — e.g. the `TelegramFallback` first-failure flag, the `SSEBroadcaster` connection dict, a per-process rate limiter. **Never** use it as the *sole* guard for state that is also in the database.

## Mechanism 2: The rest of the `asyncio` primitives family

`asyncio.Lock` is one tool in a family; reach for the others when the shape of the problem differs:

| Primitive | Models | Typical adc use |
|---|---|---|
| `asyncio.Semaphore(n)` | "At most n holders" | Already used in `src/context/warmer.py:209` (`asyncio.Semaphore(5)`) to cap concurrent context fetches. |
| `asyncio.Event` | "Broadcast a one-shot/level signal" | Signal "store is initialized" so request handlers wait for startup to finish. |
| `asyncio.Condition` | "Notify N waiters that shared state changed" | Rare; usually a `Queue` or `Event` is clearer. |
| `asyncio.Queue` | Producer/consumer serialization | Decouple producers from a single-consumer worker; the SSE broadcaster already uses one `Queue` per connection (`src/sse/broadcaster.py:27`). |
| `asyncio.Barrier` | "Wait for N coroutines to rendezvous" | Almost never needed in servers. |

These share `asyncio.Lock`'s limitations (single-process, single-loop, non-reentrant). Prefer them by *shape*: don't build a semaphore out of a lock + counter.

## Mechanism 3: Module-level / app-level mutex patterns

The parent doc rejects the module-level lock, but the rejection is overstated — module-level is correct **when the guarded state is itself process-global**. The real rule:

> **A lock must live at least as long as the state it guards, and must be reachable from every code path that touches that state.**

### Pattern A — Instance lock (use when state is per-instance)

```python
class Counter:
    def __init__(self): self._n = 0; self._lock = asyncio.Lock()
```
Right for `TelegramFallback` (one singleton instance owns the state). The parent doc's recommendation is correct *here*.

### Pattern B — Module-level lock (use when state is module-global)

```python
# src/feature/_init.py
import asyncio
_state: dict = {}
_state_lock = asyncio.Lock()   # lives with the module-global _state

async def cache_once(key, factory):
    async with _state_lock:
        if key not in _state:
            _state[key] = await factory()
        return _state[key]
```
This is the *right* pattern when the state is genuinely a module global. The parent doc's cons ("unclear ownership, hard to test") are really just "keep the lock next to the state" — which Pattern B does.

### Pattern C — App-state lock via FastAPI lifespan (preferred for the serving loop)

The cleanest place for a process-wide lock is on `app.state`, initialized in the **lifespan** handler so it is bound to the serving loop:

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI, Request
import asyncio

@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.dispatch_lock = asyncio.Lock()  # created on the serving loop
    yield
    # cleanup

app = FastAPI(lifespan=lifespan)

@app.post("/dispatch")
async def dispatch(req: Request):
    async with req.app.state.dispatch_lock:
        ...
```
This avoids the two classic module-level bugs: (1) the lock being created at import time before any loop exists, and (2) tests reusing a lock across `asyncio.run()` calls. See [FastAPI patterns](#fastapi-specific-concurrency-patterns).

### The lazy-singleton nuance (relevant to this codebase)

`get_store()` and `get_broadcaster()` use a classic lazy singleton:

```python
_store: SessionStore | None = None
def get_store(db_path=None) -> SessionStore:
    global _store
    if _store is None:
        _store = SessionStore(db_path)   # no await inside
    return _store
```

This is **safe as written** on a single loop *because the constructor does not `await`* — control never yields between the `is None` check and the assignment, so no other coroutine can interleave. **It would become unsafe** the moment init becomes async (e.g. `await store.initialize()` inside the `if`): two coroutines could both see `None`, both start init, and one store would be discarded. The fix is either (a) initialize in lifespan, or (b) guard with a lock whose `acquire()` is the first `await`. Note this is a *single-loop* argument — under multiple workers each process correctly gets its own singleton.

## Mechanism 4: `threading.Lock` + `run_in_threadpool` (when you must)

`threading.Lock` is the wrong default in async code (blocking acquire stalls the loop), but it is **required** when shared state is touched from a **threadpool** — i.e. from `def` routes, `fastapi.concurrency.run_in_threadpool`, `anyio.to_thread.run_sync`, or sync libraries that spin their own threads.

### Blocking-acquire footgun

```python
self._lock = threading.Lock()
async def handler(self):
    with self._lock:        # ❌ BLOCKS THE EVENT LOOP if contended
        ...
```
Even an uncontended `threading.Lock` acquire is cheap, but a *contended* one parks the whole event loop thread — every other request stalls. This is the core reason the parent doc rejects `threading.Lock`, and it's correct for pure-async code.

### Legitimate use: sync critical sections on the threadpool

If a function genuinely runs on a worker thread (blocking DB driver, sync CPU work), use `threading.Lock` there — it is the only lock those threads share:

```python
import threading
_lock = threading.Lock()

def _sync_compute():           # called via run_in_threadpool
    with _lock:
        ...                     # safe across threadpool threads
```

### The worst footgun: holding a `threading.Lock` across an `await`

```python
async def broken(self):
    with self._tlock:           # threading.Lock, no async release
        await self._io()        # ❌ blocks the loop *and* can deadlock
```
A `threading.Lock` cannot be released by the event loop scheduler. Never hold one across an `await`. If you need a critical section that spans `await`s, that section is on the event loop — use `asyncio.Lock`.

**Verdict:** `threading.Lock` only inside genuinely threaded code paths; never as the guard for an `async def` critical section.

## Mechanism 5: Atomic operations

> **This is the section the parent doc ("Option 4: Atomic Operations — NOT POSSIBLE") gets wrong.** Python has no CAS for *arbitrary in-memory objects*, but it has robust atomicity at the **GIL level, the SQLite level, and the filesystem level**. For any state that is persisted, these are usually a better choice than a lock.

### 5a. GIL-level atomicity (in-memory, thread layer only)

Under CPython's GIL, **a single bytecode** executes atomically with respect to other Python threads. Operations that compile to a single bytecode are atomic; **read-modify-write sequences are not**, even when they look like "one operation":

| Operation | Bytecodes | Atomic? |
|---|---|---|
| `obj.flag = True` | `STORE_ATTR` | ✅ atomic |
| `obj.n = 5` | `STORE_ATTR` | ✅ atomic |
| `d[k] = v` | `STORE_SUBSCR` | ✅ atomic |
| `lst.append(x)` | C-level call, holds GIL | ✅ atomic |
| `lst[i] += 1` (single elem) | depends | ⚠️ usually, not guaranteed |
| `obj.n += 1` | LOAD, ADD, STORE (3 bytecodes) | ❌ **NOT atomic** |
| `if not obj.flag: obj.flag = True` | check + store | ❌ **NOT atomic** (the exact first-failure race) |
| `obj.n = obj.n + 1` | same as `+=` | ❌ NOT atomic |

So a plain boolean flag set with `self.flag = True` is safe to **write** from a thread (the store is atomic), but the **check-then-set** pattern (`if not flag: flag = True`) is *not* atomic — exactly Race Condition #1 in the parent doc. The GIL is why `asyncio.Lock` is "enough" for single-worker async, but it is not a substitute for a lock or atomic op when a read-modify-write is involved.

> Reproducibility note: I tried to demonstrate a lost update on `n += 1` with 8 threads × 500k iterations and could not reproduce it on CPython 3.12 in a tight loop — the GIL switch interval let each thread run long bursts. The behavior is still **not guaranteed** (the CPython docs explicitly disclaim atomicity of compound assignments), so don't rely on it. Non-reproducibility ≠ correctness.

### 5b. SQLite atomicity (the codebase's actual shared state)

`src/session/store.py` already declares `PRAGMA journal_mode=WAL` and `PRAGMA synchronous=NORMAL`. This gives strong, real atomicity — and it is the correct mechanism for any shared state that lives in `session.db`. Key techniques:

**Transaction modes** (verified working on CPython 3.12's bundled sqlite3):

| Mode | Lock acquired | Blocks | Use when |
|---|---|---|---|
| `BEGIN DEFERRED` (default) | none until first write, then RESERVED | readers never block; writers contend lazily | Most writes; cheap but two writers can collide → `SQLITE_BUSY` |
| `BEGIN IMMEDIATE` | RESERVED immediately | other writers wait; readers proceed | Known write transaction; serializes writers cleanly |
| `BEGIN EXCLUSIVE` | EXCLUSIVE | readers *and* writers wait | Migration / batch rewrite needing total isolation |

**Atomic check-and-set / compare-and-swap via `UPDATE ... WHERE` + `rowcount`** (verified):

```python
async with aiosqlite.connect(db_path) as db:
    cur = await db.execute(
        "UPDATE first_failure SET claimed=1 WHERE id=1 AND claimed=0"
    )
    await db.commit()
    if cur.rowcount == 1:          # I won the race — exactly one writer sees this
        log_first_failure()
    # else: somebody else already claimed it; I log at DEBUG
```
The `WHERE claimed=0` makes the check-and-set a **single atomic statement**. `cur.rowcount` tells you whether you were the winner — this is a real CAS primitive, expressed in SQL. No `asyncio.Lock` needed, and it is correct across workers and restarts.

**Idempotent inserts** already used in this codebase:
- `INSERT OR IGNORE` — `create_session`, `link_intent_to_topic` (no-op if the PK / unique already exists).
- `INSERT OR REPLACE` — `set_topic_context` (clobber-and-replace is atomic).
- `UPDATE ... SET v = v + 1` — atomic increment inside one statement (the DB engine holds the row lock for the duration; no lost update, unlike Python `+=`).

**Concrete codebase race this fixes.** `SessionStore.find_or_create_topic()` (`store.py:444`) has a classic TOCTOU bug — it does a `SELECT` in one connection, then a separate `INSERT` in another, with an `await` in between:

```python
async with aiosqlite.connect(self.db_path) as db:
    ...  # SELECT: no row found
# ← another concurrent call can land here, also see "no row"
topic_id = await self.create_topic(...)   # both create → duplicate topic
```
There is no unique constraint on `(label, session_id)`, so two concurrent dispatches for the same new topic can each create a topic row. The atomic fix is a **unique index + `INSERT ... ON CONFLICT DO NOTHING ... RETURNING`** (SQLite ≥ 3.24; bundled sqlite3 on this box is 3.12-era so it supports it):

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_topic_session_label
  ON topics(label, session_id) WHERE scope='session' AND archived_at IS NULL;
```
```python
async with aiosqlite.connect(self.db_path) as db:
    cur = await db.execute(
        """INSERT INTO topics (id, label, scope, session_id, created_at, last_active)
           VALUES (?,?,?,?,?,?)
           ON CONFLICT(label, session_id) DO NOTHING
           RETURNING id""",
        (tid, label, "session", session_id, now, now),
    )
    row = await cur.fetchone()
    if row:           # we created it
        return row[0], True
    # else it already existed — fetch it
    ...
```
One statement, no lock, correct under all three concurrency layers. This is the single most useful pattern in the codebase.

### 5c. Atomic file writes (`os.replace`)

When the "store" is a file (config snapshots, the `plan.md` / notes, a JSON cache on disk), the atomic pattern is **write to a temp file in the same directory, then `os.replace`** (verified atomic on POSIX and Windows for same-filesystem renames):

```python
import os, tempfile
def atomic_write(path: str, data: bytes) -> None:
    fd, tmp = tempfile.mkstemp(dir=os.path.dirname(path), prefix=".tmp-")
    try:
        with os.fdopen(fd, "wb") as f:
            f.write(data); f.flush(); os.fsync(f.fileno())
        os.replace(tmp, path)        # atomic rename — readers never see a partial file
    except BaseException:
        os.unlink(tmp); raise
```
`os.replace` overwrites atomically; a crash leaves either the old file or the new file, never a half-written one. (Same filesystem required — `os.replace` across filesystems falls back to copy+delete and is not atomic.)

## FastAPI-specific concurrency patterns

### `async def` vs `def` endpoints — the implicit threadpool

FastAPI runs `async def` endpoints **directly on the event loop**; `def` endpoints are dispatched to a **threadpool** (Starlette's `anyio` worker, default 40 tokens). This is the single most important FastAPI concurrency fact:

- A `def` endpoint touching shared in-memory state races against **threads**, not coroutines. `asyncio.Lock` does **not** protect it — it can't even be acquired from a sync function.
- If the same shared object is touched by both an `async def` route and a `def` route (or a `BackgroundTask` that is a sync function), you have **mixed concurrency layers** and must use an atomic operation (DB) or a `threading.Lock` for the thread side — `asyncio.Lock` silently covers only half the access.

**Implication for adc:** all the first-failure / persistence paths are `async def` and single-worker, so `asyncio.Lock` is *locally* sufficient — but only as long as those constraints hold. Persisting the state atomically in SQLite (5b) is more robust and removes the constraint entirely.

### `run_in_threadpool` / `anyio.to_thread.run_sync`

For blocking calls inside `async def` (sync `httpx`, `requests`, blocking file/CPU work):

```python
from fastapi.concurrency import run_in_threadpool
result = await run_in_threadpool(blocking_fn, arg1, arg2)
```
The function runs on a worker thread — shared state it touches needs thread-layer synchronization (5a/5b/`threading.Lock`), not `asyncio.Lock`.

### Lifespan handlers — the right place to create locks

Create process-wide locks and singletons in `lifespan`, not at module import. They are then bound to the serving loop and initialized exactly once (replacing the fragile lazy-singleton pattern):

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.store = await init_store()
    app.state.telegram = TelegramFallback()      # its asyncio.Lock binds to this loop
    app.state.dispatch_lock = asyncio.Lock()
    yield
    await app.state.store.close()
```

### Injecting locks via `Depends` / `app.state`

Rather than a module global, expose the lock through `request.app.state` or a dependency so it is testable and its lifetime is tied to the app:

```python
def get_dispatch_lock(request: Request) -> asyncio.Lock:
    return request.app.state.dispatch_lock

@app.post("/dispatch")
async def dispatch(_lock: asyncio.Lock = Depends(get_dispatch_lock)):
    async with _lock: ...
```

### `BackgroundTasks` vs `asyncio.create_task`

- FastAPI `BackgroundTasks` run *after the response is sent*; a sync task runs on the threadpool, an async task on the loop. Shared-state rules above apply.
- `asyncio.create_task` runs on the loop immediately and concurrently with the handler. Anything both touch needs an `asyncio.Lock` (loop layer) or an atomic op.

### Multi-worker deployment — the hard boundary

`uvicorn --workers N` / `gunicorn -w N` runs N independent processes. **No in-memory lock, `asyncio` or `threading`, crosses this boundary.** If adc is ever scaled to multiple workers (currently it is single-process: `uvicorn src.main:app`), every in-memory lock silently stops coordinating. The only mechanisms that survive are the **atomic operations** in 5b (SQLite) or 5c (files), or an external coordinator (Redis `SETNX`, Postgres advisory locks). Designing persisted state around SQLite atomics from the start means the move to multi-worker is free.

## Decision matrix

| State lives… | Concurrency layer(s) touching it | Recommended mechanism |
|---|---|---|
| In-memory, async-only (`TelegramFallback` flag) | Event loop | `asyncio.Lock` (instance or `app.state`) — *or* persist to DB if you want durability |
| In-memory, touched by `def` routes / threadpool | Threads | `threading.Lock` for the threaded section; or move state into SQLite |
| Module-global ephemeral cache | Event loop | Module-level `asyncio.Lock` next to the data, or lifespan `app.state` lock |
| In SQLite (`session.db`) | Any / multi-worker | **SQLite atomicity**: `BEGIN IMMEDIATE`, `UPDATE…WHERE`+rowcount, `INSERT OR IGNORE`/`ON CONFLICT`. *No Python lock needed.* |
| On disk (files) | Any / multi-worker | **`os.replace` temp-then-rename** (5c) |
| Rate limiting / distributed | Multi-process / multi-host | External: Redis `SET NX EX`, or Postgres advisory lock |

### For adc-50ld's first-failure state specifically

- If the requirement is strictly "one WARNING per process lifetime, in-memory only" → `asyncio.Lock` on the singleton (the parent doc's recommendation is correct and minimal).
- If the requirement is "one WARNING per *deployment* across restarts/workers, with an accurate failure count" → store `first_failure_at` / `failure_count` as rows in SQLite and use `UPDATE…WHERE claimed=0` + `rowcount` (5b). Durable, multi-worker-safe, and no loop blocking — strictly better, at the cost of one extra column/table.

### For the SessionStore

- Keep WAL + `synchronous=NORMAL` (correct as-is).
- Fix the `find_or_create_topic` TOCTOU with a unique index + `ON CONFLICT DO NOTHING … RETURNING` (5b) — this is a real latent bug, not theoretical.
- The `get_store()` / `get_broadcaster()` lazy singletons are safe *because* their constructors don't `await`; keep them that way or move init to lifespan.

## Appendix: verified behavior (CPython 3.12.12, this box)

Ran a probe script to confirm the non-obvious claims rather than rely on memory:

| Claim | Result |
|---|---|
| `asyncio.Lock` is non-reentrant | ✅ Confirmed — second acquire in same task deadlocks (timed out via `wait_for`) |
| Reusing a lock created in one `asyncio.run()` inside another | ✅ Works on 3.12 (no `loop` binding error) — though still create on the serving loop |
| `+=` on a shared int loses updates | ⚠️ Not reproduced in a tight loop on 3.12 (GIL burst scheduling), but **not guaranteed atomic** — documented as such |
| `BEGIN IMMEDIATE` / `BEGIN EXCLUSIVE` accepted | ✅ Both work |
| `UPDATE … WHERE cond` as CAS via `rowcount` | ✅ First call `rowcount=1`, second `rowcount=0` |
| `os.replace` atomic rename | ✅ Final present, temp gone |
