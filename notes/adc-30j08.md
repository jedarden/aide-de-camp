# Intent Router Latency Verification (adc-30j08)

## Task

Verify intent router meets latency budget

## Findings

### Status: ❌ FAIL - Intent Router Exceeds Latency Budget

The intent router **significantly exceeds** its ~500ms latency budget across all test patterns.

### Budget vs. Measured

| Metric | Budget | Measured (Live Test) | Over Budget |
|--------|--------|---------------------|-------------|
| Target | ~500ms | 1,425ms - 3,568ms (p50) | 2.9x - 7.1x |
| Target | ~500ms | 1,975ms - 4,055ms (p95) | 4.0x - 8.1x |

### Test Results (2026-07-23)

**Single-intent cases (fast-path):**
- "what's the status of pbx" — p50: **1,713ms** | p95: **1,975ms** (3.4x - 4x over)
- "pull up recent logs for whisper" — p50: **1,654ms** | p95: **2,367ms** (3.3x - 4.7x over)
- "brainstorm improvements to the pipeline" — p50: **2,333ms** | p95: **3,547ms** (4.7x - 7x over)

**Multi-intent case:**
- "has the pbx web caught up and what's the state of whisper stt" — p50: **3,568ms** | p95: **4,055ms** (7x - 8x over)

### Consistency with Baseline Data

Current live test results align closely with the documented baseline (from `docs/latency-summary-table.md`):

- **Baseline p50:** 1,587-2,074ms
- **Baseline p95:** 2,527-4,301ms
- **Live test p50:** 1,425-3,568ms
- **Live test p95:** 1,975-4,055ms

The router is consistently 3-8x over its latency budget.

### Impact

This is a **demo-blocking failure**:

1. The router is a sequential hot-path stage — every millisecond of latency directly adds to the end-to-end time
2. At ~2-4s per utterance (vs. ~500ms budget), the router contributes ~1.5-3.5s of excess latency
3. Combined with synthesis latency (also 2-3x over budget), the <3s end-to-end target is unattainable

Per the plan gate (docs/plan/plan.md): "The demo cannot be scheduled until the Measured p50/p95 columns are filled from real runs... If measured p95 blows a stage's budget, either the stage gets fixed or the on-screen promise changes."

### Contributing Factors

The router latency is dominated by:
1. **ZAI proxy round-trip time** — Network overhead + LLM inference
2. **Model selection** — Currently uses Sonnet (model-class: `SONNET.value` in router.py:243)
   - Could potentially use Haiku for faster classification at reduced accuracy
3. **Prompt size** — Full segmentation prompt + urgency rules on every call
4. **Session context fetching** — Database queries for recent intents (adds I/O)

### Recommendations

To meet the ~500ms budget, consider:

1. **Model tier down** — Use Haiku-class model for routing (trade accuracy for speed)
2. **Prompt optimization** — Reduce prompt size or cache compiled prompts
3. **Fast-path caching** — Cache common single-intent classifications
4. **Eliminate session context fetch on hot path** — Move to background or defer
5. **Connection pooling** — Ensure HTTP client reuses connections to ZAI proxy
